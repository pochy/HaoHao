以下は、**Google Drive / OneDrive / Dropbox のようなファイル共有サービスを作成する際の「権限まわりの要件定義案」**です。
ZITADEL + OpenFGA のような構成を想定しても使いやすいように、**認証・認可・共有・継承・監査・例外制御**に分けて整理します。

---

# 権限管理 要件定義案

## 1. 基本方針

本サービスでは、ユーザーがファイルおよびフォルダを安全に保存・管理・共有できるように、リソース単位の細粒度な権限管理を提供する。

対象リソースは主に以下とする。

```text
Organization / Tenant
Workspace
User
Group
Folder
File
Share Link
```

権限管理では、以下を実現する。

```text
ファイル単位の閲覧・編集・削除・共有制御
フォルダ単位の閲覧・編集・削除・共有制御
フォルダから子フォルダ・子ファイルへの権限継承
ユーザー・グループ・組織単位の共有
リンク共有
外部ユーザー共有制御
監査ログ
ロック状態などによる例外制御
```

---

# 2. 権限モデル

## 2.1 基本ロール

ファイル・フォルダに対して、以下の基本ロールを定義する。

| ロール    | 説明                                                     |
| --------- | -------------------------------------------------------- |
| Owner     | 所有者。すべての操作が可能                               |
| Editor    | 編集者。閲覧・編集・一部操作が可能                       |
| Viewer    | 閲覧者。閲覧のみ可能                                     |
| Commenter | コメントのみ可能。必要であれば採用                       |
| Uploader  | アップロードのみ可能。共有フォルダ向けに必要であれば採用 |

最小構成では、まず以下の3つでよい。

```text
Owner
Editor
Viewer
```

## 2.2 操作権限

各ロールに紐づく操作権限を定義する。

| 操作                | Owner |          Editor |        Viewer |
| ------------------- | ----: | --------------: | ------------: |
| 閲覧                |    可 |              可 |            可 |
| ダウンロード        |    可 |              可 | 可 / 設定次第 |
| 編集                |    可 |              可 |          不可 |
| 名前変更            |    可 |              可 |          不可 |
| 移動                |    可 |   可 / 設定次第 |          不可 |
| コピー              |    可 |              可 | 可 / 設定次第 |
| 削除                |    可 |        条件付き |          不可 |
| 共有                |    可 |        条件付き |          不可 |
| 権限変更            |    可 | 不可 / 条件付き |          不可 |
| 所有者変更          |    可 |            不可 |          不可 |
| ロック / ロック解除 |    可 |        条件付き |          不可 |

# 3. 組織・テナント分離

## 3.1 テナント分離

各ユーザー、ファイル、フォルダ、グループ、共有リンクは、必ず特定の組織またはテナントに所属する。

```text
organization_id
workspace_id
```

を持たせる。

## 3.2 クロステナントアクセス制御

原則として、異なるテナント間のリソースアクセスは禁止する。

例外として、明示的な外部共有が有効な場合のみ、外部ユーザーまたは外部メールアドレスに共有できる。

## 3.3 テナント管理者権限

組織管理者は、組織内の以下を管理できる。

```text
ユーザー管理
グループ管理
共有ポリシー管理
外部共有可否
監査ログ閲覧
ストレージ使用量確認
リンク共有ポリシー管理
```

ただし、組織管理者が全ファイル内容を閲覧できるかどうかは、サービス設計上の重要なポリシーとして明示的に決める。

# 4. ファイル単位の権限

## 4.1 ファイル閲覧権限

ユーザーは、以下のいずれかを満たす場合にファイルを閲覧できる。

```text
ファイルのOwnerである
ファイルのEditorである
ファイルのViewerである
所属グループがファイルにViewer以上で共有されている
親フォルダからViewer以上の権限を継承している
有効な共有リンクを持っている
組織管理者として閲覧権限を持つ
```

## 4.2 ファイル編集権限

ユーザーは、以下のいずれかを満たす場合にファイルを編集できる。

```text
ファイルのOwnerである
ファイルのEditorである
所属グループがファイルにEditor以上で共有されている
親フォルダからEditor以上の権限を継承している
```

ただし、以下の場合はEditorであっても編集不可とする。

```text
ファイルがlocked状態である
ファイルが削除済み状態である
ファイルが読み取り専用状態である
組織ポリシーにより編集が禁止されている
対象ユーザーが停止中である
```

## 4.3 ファイル削除権限

ファイルを削除できるユーザーは、原則として以下とする。

```text
ファイルのOwner
削除権限を明示的に持つEditor
組織ポリシー上、削除を許可された管理者
```

削除は以下の2種類を区別する。

```text
論理削除: ゴミ箱へ移動
物理削除: 完全削除
```

Ownerまたは管理者であっても、保持ポリシーやリーガルホールドがある場合は物理削除できない。

## 4.4 ファイル共有権限

ファイルを他ユーザーへ共有できるユーザーは、以下とする。

```text
ファイルのOwner
共有権限を明示的に持つEditor
組織ポリシー上、共有を許可された管理者
```

共有可能な対象は以下とする。

```text
同一組織内ユーザー
同一組織内グループ
外部ユーザー
メールアドレス
共有リンク
```

外部共有が組織ポリシーで禁止されている場合、外部ユーザーへの共有はできない。

# 5. フォルダ単位の権限

## 5.1 フォルダ閲覧権限

ユーザーは、以下のいずれかを満たす場合にフォルダを閲覧できる。

```text
フォルダのOwnerである
フォルダのEditorである
フォルダのViewerである
所属グループがフォルダにViewer以上で共有されている
親フォルダからViewer以上の権限を継承している
有効な共有リンクを持っている
```

## 5.2 フォルダ編集権限

フォルダ編集権限には以下を含む。

```text
フォルダ名の変更
フォルダ内へのファイル追加
フォルダ内へのサブフォルダ作成
フォルダ内のファイル移動
フォルダ内のファイル編集
```

ユーザーは、以下のいずれかを満たす場合にフォルダを編集できる。

```text
フォルダのOwnerである
フォルダのEditorである
所属グループがフォルダにEditor以上で共有されている
親フォルダからEditor以上の権限を継承している
```

## 5.3 フォルダ削除権限

フォルダを削除できるユーザーは、原則として以下とする。

```text
フォルダのOwner
削除権限を明示的に持つEditor
組織ポリシー上、削除を許可された管理者
```

フォルダ削除時は、配下のファイル・サブフォルダも削除対象になるため、以下の制約を設ける。

```text
配下に削除不可ファイルがある場合、フォルダ削除を拒否する
配下にlockedファイルがある場合、フォルダ削除を拒否する
配下に保持ポリシー対象ファイルがある場合、物理削除を拒否する
```

# 6. 権限継承

## 6.1 親フォルダから子フォルダへの継承

親フォルダに設定された権限は、原則として子フォルダに継承される。

例:

```text
user:A が folder:/project の Viewer
↓
user:A は folder:/project/specs も Viewer
```

## 6.2 親フォルダから子ファイルへの継承

親フォルダに設定された権限は、配下のファイルにも継承される。

例:

```text
group:engineering が folder:/project の Editor
↓
group:engineering のメンバーは file:/project/spec.md を編集可能
```

## 6.3 継承の上書き

ファイルまたはフォルダ単位で、親フォルダからの継承を上書きできるかどうかを定義する。

推奨仕様は以下。

```text
デフォルトは継承あり
個別ファイル・フォルダで追加共有は可能
親から継承された権限を子で弱める場合は、明示的な「継承停止」機能が必要
```

継承停止を許可する場合、UI上で明確に表示する。

```text
このファイルは親フォルダの権限を継承していません
```

## 6.4 循環参照防止

フォルダ移動時に、フォルダ階層が循環しないように制御する。

禁止例:

```text
folder:A の配下に folder:B がある
folder:A を folder:B の配下へ移動する
```

# 7. グループ共有

## 7.1 グループ定義

組織内にグループを作成できる。

```text
Engineering
Sales
Finance
External Partners
```

グループには複数ユーザーを所属させられる。

## 7.2 グループへの共有

ファイルまたはフォルダをグループに共有できる。

```text
group:engineering に folder:/project を Editor として共有
group:sales に file:/proposal.pdf を Viewer として共有
```

## 7.3 グループメンバー変更時の反映

グループにユーザーを追加した場合、そのグループに共有されているリソースへ即時または短時間でアクセス可能になる。

グループからユーザーを削除した場合、そのユーザーはグループ経由のアクセス権を失う。

## 7.4 グループ共有の監査

以下のイベントを監査ログに記録する。

```text
グループ作成
グループ削除
グループメンバー追加
グループメンバー削除
グループへのファイル共有
グループへのフォルダ共有
グループ共有解除
```

---

# 8. ユーザー共有

## 8.1 個別ユーザー共有

Ownerまたは共有権限を持つユーザーは、ファイル・フォルダを個別ユーザーに共有できる。

共有時に指定可能な権限は以下。

```text
Viewer
Editor
Owner
```

Owner移譲は特別な操作として扱い、確認操作を必須とする。

## 8.2 共有招待

未登録ユーザーへ共有する場合、メールアドレス宛に招待を送信できる。

招待には以下を持たせる。

```text
招待先メールアドレス
対象リソース
付与権限
有効期限
招待者
状態: pending / accepted / expired / revoked
```

## 8.3 外部ユーザー共有

外部ユーザーへの共有は、組織ポリシーに従う。

ポリシー例:

```text
外部共有を許可する
外部共有を禁止する
特定ドメインのみ許可する
特定ドメインを禁止する
管理者承認後のみ許可する
```

---

# 9. リンク共有機能

## 9.1 基本仕様

ファイルまたはフォルダに対して共有リンクを作成できる。

共有リンクには以下の種類を持たせる。

```text
リンクを知っている全員が閲覧可能
同一組織内のユーザーのみ閲覧可能
指定された外部ユーザーのみ閲覧可能
無効化済みリンク
```

## 9.2 リンクを知っている人はViewer

共有リンクが有効な場合、リンクを知っているユーザーは対象リソースをViewerとして閲覧できる。

ただし、以下の条件を満たす必要がある。

```text
リンクが有効である
リンクの有効期限が切れていない
リンクが削除・無効化されていない
組織ポリシーでリンク共有が許可されている
対象リソースが削除済みでない
```

## 9.3 有効期限つきリンク

共有リンクには有効期限を設定できる。

```text
expires_at
```

有効期限を過ぎたリンクではアクセスできない。

## 9.4 パスワードつきリンク

共有リンクにはパスワードを設定できる。

保存時は平文ではなく、ハッシュ化して保存する。

```text
password_hash
```

パスワード入力に一定回数失敗した場合、レート制限または一時ブロックを行う。

## 9.5 社外共有禁止

組織ポリシーで社外共有禁止が設定されている場合、以下を禁止する。

```text
外部メールアドレスへの共有
外部ユーザーへの招待
anonymous link の作成
public link の作成
```

既存の外部共有については、ポリシー変更時に以下のいずれかを選択できる。

```text
即時無効化
新規共有のみ禁止
管理者に警告表示
```

## 9.6 ダウンロード禁止

共有リンクまたはViewer権限に対して、ダウンロード禁止を設定できる。

```text
can_download = false
```

この場合、ユーザーはブラウザ上でプレビューできるが、ダウンロードボタンは表示されない。

ただし、完全な情報漏洩防止ではないため、UI上または管理者向けに以下を明示する。

```text
ダウンロード禁止は操作上の制限であり、スクリーンショット等の取得を技術的に完全には防止できない
```

## 9.7 リンク共有の権限レベル

リンク共有で許可できる権限は、原則Viewerのみとする。

必要に応じて以下を追加する。

```text
Viewer
Commenter
Editor
```

ただし、匿名Editorリンクは危険度が高いため、初期リリースでは非推奨。

# 10. 共有解除

## 10.1 個別共有解除

Ownerまたは共有管理権限を持つユーザーは、個別ユーザー・グループに付与された権限を解除できる。

解除対象:

```text
ユーザー共有
グループ共有
外部ユーザー共有
共有リンク
招待
```

## 10.2 継承権限の解除

親フォルダから継承された権限は、子リソース側で直接解除できない。

解除するには以下のいずれかを行う。

```text
親フォルダ側の共有を解除する
子リソース側で継承を停止する
子リソースを別フォルダへ移動する
```

## 10.3 共有解除時の監査

共有解除時には以下を記録する。

```text
実行者
対象リソース
解除された対象ユーザー/グループ/リンク
解除前の権限
実行日時
IPアドレス
User-Agent
```

# 11. ロック状態

## 11.1 ファイルロック

ファイルにはlocked状態を設定できる。

```text
locked = true
```

locked状態のファイルでは、Editorであっても以下の操作を禁止する。

```text
編集
上書きアップロード
名前変更
移動
削除
共有変更
```

Ownerまたは特定権限を持つユーザーのみ、ロック解除できる。

## 11.2 ロック理由

ロックには理由を持たせる。

```text
manual_lock
system_lock
legal_hold
approval_pending
retention_policy
```

## 11.3 ロック中の閲覧

locked状態でも、閲覧権限があるユーザーは閲覧可能とする。

ただし、組織ポリシーによりダウンロードは禁止できる。

# 12. 閲覧可能ファイル一覧

## 12.1 一覧取得要件

ユーザーは、自分が閲覧可能なファイル・フォルダ一覧を取得できる。

一覧には以下を含める。

```text
自分が所有するファイル
自分に直接共有されたファイル
所属グループ経由で共有されたファイル
親フォルダから権限継承されたファイル
共有リンク経由でアクセス可能なファイル
```

## 12.2 フォルダ表示

フォルダを開いたとき、ユーザーが閲覧可能な子ファイル・子フォルダのみ表示する。

権限がないリソースは一覧に表示しない。

## 12.3 検索

検索結果には、ユーザーが閲覧可能なファイル・フォルダのみ表示する。

```text
キーワード検索
ファイル種別検索
所有者検索
更新日時検索
共有状態検索
```

すべての検索結果に権限フィルタを適用する。

## 12.4 パフォーマンス

大量ファイルを扱うため、以下を考慮する。

```text
権限判定のキャッシュ
ListObjects系APIの利用
DB側インデックス
非同期インデックス更新
ページネーション
```

# 14. 組織ポリシー

組織単位で以下のポリシーを設定できる。

## 14.1 共有ポリシー

```text
外部共有を許可する / 禁止する
リンク共有を許可する / 禁止する
匿名リンクを許可する / 禁止する
パスワードなしリンクを許可する / 禁止する
リンク有効期限を必須にする
最大有効期限を設定する
Editorによる再共有を許可する / 禁止する
```

## 14.2 ダウンロードポリシー

```text
Viewerのダウンロードを許可する / 禁止する
外部ユーザーのダウンロードを禁止する
共有リンク経由のダウンロードを禁止する
特定ファイル種別のダウンロードを禁止する
```

## 14.3 削除ポリシー

```text
Editorによる削除を許可する / 禁止する
完全削除はOwnerのみ許可
完全削除は管理者のみ許可
ゴミ箱保持期間を設定する
```

## 14.4 セキュリティポリシー

```text
MFA必須
外部共有時に警告表示
特定ドメインのみ共有許可
IP制限
デバイス制限
セッション期限
```

# 15. ゴミ箱・削除・復元

## 15.1 論理削除

ファイル・フォルダ削除時は、原則としてゴミ箱へ移動する。

```text
deleted_at
deleted_by
```

を記録する。

## 15.2 復元

Ownerまたは復元権限を持つユーザーは、ゴミ箱から復元できる。

復元時には、元の親フォルダが存在するか確認する。

存在しない場合は、復元先を選択させる。

## 15.3 完全削除

完全削除は、以下の条件を満たす場合のみ可能とする。

```text
完全削除権限がある
保持ポリシー対象ではない
locked状態ではない
リーガルホールド対象ではない
```

# 16. 競合・例外・優先順位

## 16.1 権限の優先順位

権限判定は原則として、より強い権限を優先する。

```text
Owner > Editor > Viewer
```

ただし、明示的な禁止ルールがある場合は、許可より禁止を優先する。

例:

```text
ファイルがlockedならEditorでも編集不可
組織ポリシーで外部共有禁止ならOwnerでも外部共有不可
保持ポリシー対象ならOwnerでも完全削除不可
```

## 16.2 明示的Deny

必要であれば、明示的Denyを導入する。

```text
user:A は folder:/project のViewer
ただし file:/project/secret.pdf は閲覧不可
```

ただし、明示的Denyは権限モデルを複雑にするため、MVPでは非推奨。

MVPでは、以下を推奨する。

```text
継承停止
個別共有
組織ポリシーによる禁止
```

# 17. API要件

## 17.1 権限チェックAPI

各操作前に権限チェックを行う。

例:

```text
canViewFile(userId, fileId)
canEditFile(userId, fileId)
canDeleteFile(userId, fileId)
canShareFile(userId, fileId)

canViewFolder(userId, folderId)
canEditFolder(userId, folderId)
canDeleteFolder(userId, folderId)
canShareFolder(userId, folderId)
```

## 17.2 共有API

```text
POST /files/:fileId/shares
DELETE /files/:fileId/shares/:shareId

POST /folders/:folderId/shares
DELETE /folders/:folderId/shares/:shareId
```

## 17.3 共有リンクAPI

```text
POST /files/:fileId/share-links
PATCH /share-links/:shareLinkId
DELETE /share-links/:shareLinkId

POST /folders/:folderId/share-links
PATCH /share-links/:shareLinkId
DELETE /share-links/:shareLinkId
```

## 17.4 権限一覧API

```text
GET /files/:fileId/permissions
GET /folders/:folderId/permissions
```

返却内容には、直接付与された権限と継承された権限を区別して含める。

```json
{
  "direct": [],
  "inherited": []
}
```

# 19. MVPで入れるべき範囲

最初のMVPでは、以下を優先するのがおすすめです。

```text
組織/テナント分離
Owner / Editor / Viewer
ファイル単位の閲覧・編集・削除
フォルダ単位の閲覧・編集・削除
親フォルダから子ファイルへの権限継承
ユーザー共有
グループ共有
共有解除
閲覧可能ファイル一覧
基本的な監査ログ
ファイルlocked時の編集禁止
```

リンク共有は便利ですが、セキュリティリスクが高いため、MVPでは以下の最小構成がよいです。

```text
リンクを知っている人はViewer
有効期限つきリンク
リンク無効化
ダウンロード禁止
```

パスワードつきリンク、外部ドメイン制限、承認フローなどは次フェーズでもよいです。

# 20. 推奨する権限チェックの考え方

権限判定は、毎回以下の順で行う。

```text
1. ユーザーが認証済みか確認する
2. 対象リソースが存在するか確認する
3. テナント境界を確認する
4. リソース状態を確認する
   - deleted
   - locked
   - archived
   - retention
5. 組織ポリシーを確認する
6. ユーザー/グループ/継承/リンク共有の権限を確認する
7. 許可または拒否を返す
8. 必要に応じて監査ログを記録する
```
